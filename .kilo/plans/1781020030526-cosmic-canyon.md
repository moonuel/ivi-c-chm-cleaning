# Plan: Portable SQLite IVI-C Index

## Goal
Replace the unstable Whoosh-style index with a single portable SQLite artifact that an LLM agent and future C++ consumers can query directly, while preserving the current CLI surface and search semantics.

## Source of truth
- HTML corpus: `data/extracted/Html/`
- Skill workflow: `.kilo/skills/ivi-chm/SKILL.md`
- Current CLI entry points: `parse`, `index`, `search`
- Current result contract: `symbol`, `kind`, `title`, `summary`, `source_path`, `snippet`, `matched_on`

## Required behavior
1. `index` writes one SQLite file, not a directory index.
2. `search` reads that same SQLite file.
3. Lookup order stays:
   - exact `symbol` / `path_id`
   - normalized alias match
   - full-text fallback
4. `source_path` remains the canonical navigation/citation field.
5. Search output shape stays stable for existing tests and downstream callers.
6. The skill docs must direct the agent to the portable SQLite artifact and the exact-first workflow.

## Implementation plan
1. Redesign `src/ivi_chm/indexer.py` around a simple SQLite schema.
   - Keep canonical records in `documents`.
   - Store normalized lookup columns for exact/alias resolution.
   - Add a separate full-text strategy that is easy to rebuild and does not depend on fragile external-content maintenance.
   - Make rebuild deterministic by deleting/recreating the target DB file before indexing.
2. Update the CLI wiring in `src/ivi_chm/cli.py`.
   - Default to `.ivi-chm-index.sqlite3`.
   - Treat the second argument as an index file path consistently for both `index` and `search`.
3. Align test fixtures in `tests/`.
   - Make fixture names and paths reflect a file-based index.
   - Preserve current behavioral assertions for exact, alias, and full-text search.
4. Update documentation.
   - `README.md` should describe the portable SQLite artifact and the exact/alias/full-text workflow.
   - `.kilo/skills/ivi-chm/SKILL.md` should point to the new artifact path and the same lookup order.
5. Verify with focused tests.
   - CLI parse/index/search coverage.
   - Search quality and metrics coverage.
   - Parser contract coverage if indexing depends on it.

## Design choice
Prefer the simplest reliable SQLite design over a clever FTS integration. If FTS5 is used, keep it isolated and rebuildable; if not, use indexed normalized columns plus a straightforward text-search fallback that remains deterministic.

## Acceptance criteria
- `index` succeeds from the extracted HTML corpus and produces one `.sqlite3` file.
- `search` returns the same semantic ordering as before.
- The skill workflow points the agent to the portable index and `source_path` citations.
- Existing search-quality tests pass without changing their intent.
- No runtime Python dependency is required for consumers of the index artifact.
