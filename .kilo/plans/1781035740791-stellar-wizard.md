# Plan: Benchmark ivi Docs Skill vs Grep

## Goal
Measure whether the `ivi-chm-docs` skill finds and answers documentation questions from the extracted IVI-C HTML corpus better than a plain grep-style baseline.

## Benchmark Question Set
- Create a frozen set of 100 questions sourced from `data/extracted/Html/`.
- Split the set evenly across four categories:
  - 25 exact lookup questions
  - 25 section-location questions
  - 25 multi-hop synthesis questions
  - 25 paraphrase/semantic-match questions
- Ensure each question has one or more gold source files and an accepted answer span or span set.
- Keep a small negative set if needed for sanity checks, but do not mix it into the main score.

## Corpus Freeze
- Use one fixed snapshot of the extracted HTML corpus.
- Record the corpus hash or build timestamp so results are reproducible.
- Do not allow either system to inspect files outside the frozen corpus.

## Benchmark Adapters
1. Grep baseline
- Implement a deterministic search adapter over the HTML corpus.
- Prefer simple text lookup using `rg`-style behavior and fixed parsing rules.
- Return the top matching file paths, line ranges, and quoted snippets.
- Do not add semantic reranking or LLM interpretation.

2. ivi docs skill
- Run the `ivi-chm-docs` workflow against the same question set.
- Capture the final answer, cited `source_path` values, and any quoted evidence.
- Treat the skill output as the candidate system under test, not a rewritten baseline answer.

## Question Schema
Store each benchmark item in a machine-readable JSON file.

```json
{
  "id": "q-001",
  "category": "multi_hop",
  "question": "How does feature X behave when Y is disabled?",
  "gold": {
    "source_files": ["data/extracted/Html/A.html", "data/extracted/Html/B.html"],
    "answer_spans": [
      {"file": "data/extracted/Html/A.html", "start": 120, "end": 148},
      {"file": "data/extracted/Html/B.html", "start": 77, "end": 91}
    ]
  },
  "acceptance": {
    "must_contain": ["behavior", "disabled"],
    "must_not_contain": ["unsupported speculation"]
  }
}
```

## Execution Protocol
- Run both systems against the same questions in the same order.
- Use the same per-question budget for both systems.
- Record for every run:
  - question id
  - system name
  - timestamp
  - runtime
  - tool-call count
  - retrieved sources
  - final answer text
  - evidence snippets
  - failure reason if any
- Keep the benchmark runner deterministic enough to rerun on later corpus snapshots.

## Budgets
- Time limit: 60 seconds per question per system.
- Tool-call limit: 10 per question per system.
- Output limit: 20k characters per question per system.
- If a system exceeds budget, mark the run as a failure rather than truncating silently.

## Scoring
Score each answer on a 0-5 scale.
- 0: wrong or no answer
- 1: partially relevant but not usable
- 2: correct topic, wrong detail
- 3: correct answer with weak evidence
- 4: correct answer with correct source citation
- 5: correct answer with correct source and concise, complete evidence

Also compute these automatic metrics:
- source retrieval accuracy
- evidence span overlap
- exact/normalized answer match where applicable
- latency
- tool-call count
- failure rate

## Human Review
- Review all borderline cases where the automatic score is ambiguous.
- Mark each answer as correct, partially correct, or incorrect.
- Flag unsupported speculation separately from factual errors.
- For synthesis questions, judge whether the answer combines the needed pages correctly.

## Analysis Output
Produce a report with:
- overall average score by system
- per-category scores
- median latency by system
- source retrieval precision and recall
- top 10 failures for grep
- top 10 failures for the skill
- paired examples where grep wins
- paired examples where the skill wins

## Win Criteria
Treat the skill as better if either of the following holds:
- at least 15% higher average score overall, or
- at least 20% higher score on semantic and multi-hop questions, with no more than 5% regression on exact lookup

## Implementation Steps
1. Freeze the corpus snapshot and define the benchmark JSON schema.
2. Build the question set and gold annotations from the HTML corpus.
3. Implement the grep baseline adapter and its deterministic snippet extraction.
4. Implement the skill-run adapter and transcript capture.
5. Add scoring and aggregation code.
6. Run the benchmark on the frozen corpus and inspect the first report.
7. Tune only the benchmark harness, not the systems under test, unless a harness bug is found.
8. Publish the final comparison report.

## Risks / Decisions
- Need to decide whether the skill adapter is run manually, via scripted Kilo sessions, or through captured transcripts.
- Need to decide whether exact line spans are available for all gold answers or whether some questions should use file-level evidence only.
- Need to keep the baseline honest: no semantic reranking, no LLM assistance, and no hidden post-processing beyond snippet extraction.
