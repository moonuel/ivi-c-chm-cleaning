# Portable Index Plan

## Goal
Replace the current Python/Whoosh search index with a portable, rebuildable SQLite-based index that can be consumed from C++ without requiring a Python runtime, while keeping the coding-agent workflow accessible through a Kilo skill.

## Scope
- Preserve the existing HTML parsing pipeline.
- Replace the on-disk search index backend.
- Keep exact-first lookup and keyword/full-text fallback behavior.
- Make the index artifact portable across repos.
- Document the index as a Kilo skill for agent consumption.
- Leave a clear path for future semantic/vector search.

## Proposed Architecture
- Use Python only as the offline index builder.
- Store extracted docs in a single SQLite database file.
- Use normal tables for canonical fields and FTS5 for text search.
- Store schema/version metadata in the database.
- Query from C++ through SQLite directly.

## Implementation Steps
1. Define the SQLite schema.
- `documents` table for canonical fields: `symbol`, `kind`, `path_id`, `title`, `summary`, `abstract`, `prototype`, `remarks`, `keywords`, `function_tree_node`, `see_also`, `source_path`, and normalized lookup columns.
- `aliases` table if needed for multi-value alias lookups.
- `documents_fts` virtual table for full-text search over the descriptive fields.
- `meta` table for `schema_version`, `build_timestamp`, and optional future embedding metadata.

2. Replace the Whoosh indexer.
- Rewrite `src/ivi_chm/indexer.py` to build/query SQLite instead of Whoosh.
- Keep exact symbol and path lookup as the first match path.
- Keep alias lookup as the second match path.
- Keep full-text fallback using FTS5.
- Preserve the current result shape used by the CLI (`symbol`, `kind`, `title`, `summary`, `source_path`, `snippet`, `matched_on`).

3. Update the CLI contract.
- Keep `parse`, `index`, and `search` subcommands.
- Keep the default index path, but change it to the SQLite file name/location.
- Ensure existing tests and callers continue to work with minimal change.

4. Update packaging/build flow.
- Replace the Whoosh-specific assumptions in the build path.
- Keep `build_exe.ps1` working for the CLI if the executable remains useful.
- If a standalone executable is still desired, ensure it only bundles the index builder/search CLI, not runtime search dependencies.

5. Add/adjust tests.
- Verify exact symbol search.
- Verify alias search.
- Verify full-text fallback.
- Verify rebuild behavior when the schema version changes.
- Verify database output is deterministic enough for regeneration.

6. Add a Kilo skill for portability.
- Document canonical DB path and file name.
- Document lookup order and citation rules.
- Document the schema versioning strategy.
- Tell agents to use SQLite directly from C++ and to prefer exact symbol lookup before broader search.

7. Prepare for future vector search.
- Reserve a metadata field for embedding model/version.
- Keep a separate table or sidecar path for vectors if/when semantic search is added.
- Do not introduce runtime vector dependencies in this phase.

## Acceptance Criteria
- The index can be rebuilt from the extracted HTML into a portable SQLite file.
- Search behavior remains exact-first with text fallback.
- The end user can consume the index from C++ without Python installed.
- The coding agent can discover and use the index through a Kilo skill.
- Schema changes are handled by versioning and rebuilds.

## Risks / Decisions
- Need to confirm whether the environment SQLite build supports FTS5.
- Need to decide whether aliases live in a separate table or as a normalized column set.
- Need to decide whether the portable artifact should be checked into the repo or generated in release/build steps.
