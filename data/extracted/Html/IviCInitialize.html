Initializing the IVI-C Driver

The application you write must communicate with the driver:

  • Through the use of KtNA in function call names
  • By directly referencing KtNA.h and KtNA.lib (as shown in the example)

This section describes the use and parameters of the Initialize functions independent of a programming ADE. The example in this section shows how to use the Initialize function with C++. If you are using an ADE, we recommend you read this section first, then go to the appropriate section in ADE Programming Details for information on calling the functions with ADE-specific syntax.

Calling the Initialize Function

To initialize your driver, invoke the Initialize or Initialize With Options functions directly on the KtNA driver. This gives you access to the driver and the instrument's capabilities. The simplest example looks like this:

  ViSession session;
  ViStatus status = KtNA_InitWithOptions("MyVisaAlias", VI_TRUE, VI_TRUE, "", &session);

The initialize functions use the following parameters (described in the next section) to control fundamental aspects of the driver's behavior:

  • ResourceName
  • IdQuery
  • Reset
  • OptionsString (supported for initWithOptions only)
  • Vi (unique identifier for an IVI session; always set to true)

Initialize Function Parameters

The initialize functions offer a variety of options and parameters that control fundamental aspects of the driver's behavior. This section describes them and explains how they impact driver operation.

Resource Name

The ResourceName parameter is either:

  • VISA resource descriptor – VISA resource descriptors include all of the information needed to specify the location of the instrument on the specified I/O interface. The table below documents the format of resource descriptors for various I/O interfaces. VISA aliases and IVI logical names both resolve to VISA resource descriptors.
  • VISA alias – A string (referrred to as MyVisaAlias) that can be used instead of a resource descriptor in VISA programs. In the driver, aliases are passed directly to VISA, which resolves the alias to the VISA resource descriptor. Using VISA aliases (rather than hard-coded resource descriptors) makes your programs more portable, and Keysight recommends them for readability and flexibility. You can define VISA aliases for your instruments in Keysight IO Libraries Connection Expert. Refer to IO Libraries Connection Expert documentation for instructions.
  • IVI logical name – A name for an instrument defined in the IVI Configuration Store. Refer to the documentation for your Config Store editor for details.

IdQuery

When enabled, the driver queries the instrument model and compares it with a list of instrument models supported by the driver. If the model is not supported, initialization fails with the IVI_ERROR_ID_QUERY_FAILED error code.

Note: Most modular instruments ignore this setting and always validate that they are communicating with a supported instrument.

Reset

When enabled, the instrument is reset when the driver is initialized. If the reset fails, initialization fails with the IVI_ERROR_RESET_FAILED error code.

Note: Most modular instruments ignore this setting and start from a known reset state after initialization.

Option String

Option string allows you to pass optional settings to the driver.

The string is composed of comma-separated assignment operations in the form "Operation=value". Spaces are ignored except as separators. For example:

   "Simulate=true, DriverSetup= Model=E5080A", &session)

The following table describes the operations and values that may be used.

Option Name Description Default
QueryInstrStatus Specifies whether the IVI driver queries the instrument status at the end of each user operation. Querying the instrument status is very useful for debugging. After validating the program, the user can set this attribute to False to disable status checking and maximize performance. The user specifies this value for the entire IVI driver session. false
Simulate Specifies whether the IVI driver simulates instrument driver I/O operations. If simulation is enabled, the driver functions do not perform instrument I/O. For output parameters that represent instrument data, the driver functions return simulated values. false
DriverSetup Specifies additional settings supported by the driver, but not defined by IVI. ""

Note:
This driver does not support the Cache, InterchangeCheck, RangeCheck, and RecordCoercions options. These all default to false.

If the resource name is a logical name defined in the IVI Configuration Store, then:

  • Any optional settings that are specified in the IVI Configuration Store are loaded first.
  • The options parameter is processed next. If there are conflicts, the values specified in the options parameter override those from the Config Store. Any setting that is not specified defaults to its default value. The default values for IVI defined settings are specified by the IVI Foundation, while the driver defines the default values for driver specific settings.

DriverSetup

DriverSetup is a special token whose subsequent assignments are specific to the driver, but not defined by IVI. When the option string parameter contains the DriverSetup attribute, the Initialize function assumes that everything following 'DriverSetup=' is part of the assignment. Spaces are ignored, as the "Operation=Value" assignments are separated by commas.

The following settings are supported by the KtNA driver.

Option Name Description Default
Model Instrument model to use during simulation. Must be one of the Supported Models strings listed on the IVI Compliance Information page. E5080A
Trace If true, append a trace log of all driver calls to a text file in the same directory as the application executable accessing the driver. Additional tracing parameters may be set using the driver's SystemTracing functions and SYSTEM_TRACING attributes. false
TraceArraySizeMax Specifies the maximum number of elements to log for array parameters. 10

Example: Accessing the Driver Directly with Simulation

The following code:

  • Initializes the driver in simulation mode (Simulate=true)
  • Specifies the model to simulate (DriverSetup=Model=KtNA)

The following code initializes the driver in simulation mode and specifies the model to simulate. The VISA alias "MyVisaAlias" must be created before running this example if you do not specifiy Simulate=true.

C++ Copy Code
#include "KtNA.h" // Specific driver header file

void main()
{
  ViStatus status;
  ViSession session;
  
  // Initialize the driver in simulation mode.  Resource name is ignored when Simulate=true.
  // IdQuery and Reset options are enabled.
  // "MyVisaAlias" is a VISA alias for your instrument.
  status = KtNA_InitWithOptions("MyVisaAlias", VI_TRUE, VI_TRUE, "Simulate=true, DriverSetup= Model=E5080A", &session);
  
  // ... call KtNA driver functions
  
  status = KtNA_close(session);
}
