IVI Configuration Store

The IVI Configuration Store (referred to as Config Store) serves as a central repository for driver registration information. The Config Store is:

  • Part of the IVI Shared Components
  • An XML file named IviConfigurationStore.xml that is installed in <IviInstallDir>\Data\ (typically C:\Program Files\IVI Foundation\IVI\Data)

How the Config Store is Used

IVI drivers are required to populate the Config Store with one entry: the software module. This entry contains:

  • The instrument driver DLL
  • Any repeated capability physical names
  • Additional IVI compliance information
  • A variety of other items

Note: The software module is created by the KtNA installer and removed by the uninstaller. End users should never modify the software module entry.

End users can add information to the Config Store to describe how they intend to:

  • Use IVI drivers in their system
  • Specify logical names for IVI drivers
  • Specify virtual names for channels

If you want to:

  • Access only the driver's instrument-specific functionality, no other entries in the Config Store are required.
  • Use the driver in an interchangeable program, you must edit the Config Store (see the following section). Instrument vendors and other third parties provide graphical tools for conveniently editing the Config Store.

Editing the IVI Configuration Store

Note:

Although the IVI Configuration Store is a human readable XML file, the IVI Foundation strongly discourages you from directly editing this file. The Config Store file is highly self-referential and easily corrupted if directly modified. Modifications are typically done with either:

  • A third-party utility (search IVI Configuration Store Editor to see options)
  • The  IVI Configuration Server, a programmatic interface to the Config Store that is included in the IVI shared components

Note: End users should never modify the software module entry.

To write test programs that allow for interchangeability, you must create three additional entries (described below) in the Config Store: Hardware Asset, Driver Session, and Logical Name. Refer to the documentation for your Config Store editor for detailed instructions on modifying these items.

Hardware Asset

The hardware asset identifies a specific piece of instrument hardware. Its main purpose is to house the I/O resource descriptor used to communicate with the device.

Driver Session

The driver session associates the software module with the hardware asset and specifies such things as:

  • Driver initialization values
  • Mappings between any repeated capability physical names and virtual names (which are used by client programs to access repeated capabilities in an interchangeable fashion; see Repeated Capabilities for details)

Users create a separate driver session for each unique combination of driver software, instrument hardware, and initialization settings. You can have multiple driver sessions for the same software module, if, for instance, the driver is controlling several instances of the same instrument at different GPIB addresses.

Logical Name

The logical name allows you to associate a name you choose (such as MyDriverSession1) with a driver session. Because the driver session is tied to a particular driver and instrument, a level of indirection is needed to abstract these details in interchangeable applications. The logical name—a simple string identifier for a driver session—serves as this extra level of abstraction.

Your program must use the logical name to be interchangeable. Applications load IVI-C-specific drivers by passing a logical name to the driver's Initialize function. No instrument-specific details exist in the client program. Swapping instruments in such a system involves merely associating the logical name in the Config Store with a different driver session (in other words, a different driver-instrument combination).
