Instrument Interchangeability

Instrument interchangeability allows test system engineers to replace one instrument in a test system with another, without modifying the code. Code written for an IVI-C class driver is interchangeable because:

  • An instrument-independent logical name identifies the instrument driver to be used
  • The function calls have no instrument-specific references because only IVI class-compliant functionality is used

The IVI Foundation supports interchangeability through IVI specifications for instrument classes, as well as a standard API for programming instruments in a particular class. Using this standard, test engineers can build hardware-independent test systems.

This topic provides an overview of interchangeability. For more information, see Interchangability in the IVI Foundation's Getting Started Guide.

About IVI Instrument Classes

To enable interchangeability, the IVI Foundation creates IVI instrument classes that define the base class capabilities and class extension capabilities for some of the most popular types of instruments. Although many types of test and measurement instruments offer a variety of functionality, most of them naturally fall into distinct categories. These instruments are often categorized based on the type of measurements they perform or the type of output they produce.

For each instrument class, the IVI Foundation publishes a class specification that:

  • Details the set of functions and attributes that are required for compliance
  • Prescribes the exact name of each function, along with the parameter list and the API hierarchy, as well as other behavior details

An IVI driver that complies with a class specification is said to be class-compliant. The IVI Foundation currently has specifications for 13 instrument classes (see the IVI Specifications for details):

  • IVI-4.1: IviScope class
  • IVI-4.2: IviDmm class
  • IVI-4.3: IviFgen class
  • IVI-4.4: IviDCPwr class
  • IVI-4.5: IviACPwr class
  • IVI-4.6: IviSwtch class
  • IVI-4.7: IviPwrMeter class
  • IVI-4.8: IviSpecAn class
  • IVI-4.10: IviRFSigGen class
  • IVI-4.12: IviCounter class
  • IVI-4.13: IviDownconverter class
  • IVI-4.14: IviUpconverter class
  • IVI-4.15: IviDigitizer class

Note: For information on this driver's IVI instrument class support (if any), see IVI Compliance.

How Interchangeability Works

Instrument Interchangeability refers to the ability to exchange an instrument of one class in a system with a different instrument from that same class. IVI interchangeability does not include exchanging instruments from different instrument classes.

Client applications achieve interchangeability by implementing code that meets at least the following two conditions:

  • Uses only functions and attributes from the IVI class specification
  • Has no references to a specific IVI instrument driver

Because IVI drivers can contain functionality beyond the class specification (known as instrument-specific functionality), you must ensure that your application uses only functionality from the IVI specification if your program is to be interchangeable. Only the IVI-defined functionality is guaranteed to be supported by a class-compliant driver.

Note:

IVI interchangeability does not guarantee that instrument behavior is 100% interchangeable. As a general principle, standard programming interfaces insure that syntax is interchangeable, but only define "loose" semantics – that is, limited semantics that define the most important elements of behavior, but not all. For example, some instruments may perform more quickly than others, or have different couplings between properties, or support different ranges. Never assume that IVI Interchangeability allows you to simply substitute one instrument for another without thoroughly testing the behavior of each instrument in the system.

IVI-C Interchangeability

IVI-C interchangeability requires the use of a special third-party component called a class driver.

Important:

Neither Keysight nor the IVI Foundation distribute class drivers. To write an interchangeable application for your instrument, you must obtain a NoClass class driver from a third-party vendor (search IVI-C Class Driver for options).

The class driver exposes the functions defined in a particular IVI instrument class. All class driver functions have generic names, so that client applications have no driver-specific references. For example, rather than naming a function KtNA_Configure, which is obviously device-specific, the class driver exposes the function with the name IviDmm_Configure.

IVI-C client applications link to the class driver and call into the class driver's Initialize function using a logical name for the ResourceDescriptor parameter. The logical name identifies the IVI-C instrument driver that the class driver should load. After the instrument driver is loaded, subsequent function calls on the class driver are delegated to the corresponding function in the instrument driver. In this way, IVI-C client applications can be written generically against a class driver and work with new instrument drivers without modification.

Base Capabilities and Extension Groups

Each IVI class specification partitions IVI-defined functionality into two groups: base capabilities and extension groups.

Base Capabilities Group

The base capabilities group:

  • Contains functionality that all IVI drivers must implement if they wish to claim conformance with the instrument class in question
  • Represents the "least common denominator" of functionality and is the level of capability most commercial instruments are expected to support

For example, the IviDmm base capabilities group contains functions for range, resolution, and triggering. Most commercial DMMs possess at least these capabilities, so any IVI driver that claims to be IviDmm-compliant must support this group of functionality.

Extension Group

The extension group:

  • Defines functionality that is found in many instruments of a particular class but is not as common as base capabilities functionality
  • Can be thought of as providing standard definitions for optional functionality
  • Enables more advanced instruments to expose functionality in an interchangeable manner, while still allowing simpler instruments to be class-compliant

For example, because not all DMMs can measure temperature, the IviDmm class specification has an IviDmmTemperature extension group to define temperature measurement capabilities.

Simply put, an extension group tells the driver developer that "you do not have to provide these capabilities in your IVI driver, but if you do, you must do it according to the IVI definition." An IVI driver can be class-compliant without supporting any of the extension groups in its instrument class. All IVI drivers expose a GroupCapabilities attribute that returns a comma-separated list of the extension groups supported by the driver. Using this attribute, client applications and tools can programmatically determine the IVI driver's optional functionality.
