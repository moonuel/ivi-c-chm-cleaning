IVI Driver Inherent Capabilities

IVI inherent capabilities are a set of functions and attributes that all IVI drivers must support, regardless of the IVI instrument class they support, and even it they support no class. This ensures that instrument users have a standard, recognizable way to perform basic operations with an IVI driver. Users who have programmed with any type of IVI driver will immediately know how to perform basic tasks with any new IVI driver encountered.

For more information, see IVI-3.2: Inherent Capabilities in the IVI Specifications.

Required Driver Tasks

Tasks that fall under the inherent capabilities specification include:

  • Initializing the driver
  • Querying errors from the driver
  • Enabling standard driver features, such as simulation, state caching, and range checking
  • Identifying the capabilities of the driver, such as the supported instrument classes, capability groups.
  • Reading driver and instrument identification information
  • Performing instrument self-test and reset operations

For more information, see Initializing the IVI-C Driver, Inherent IVI Attributes and Utility Functions.

Standard Driver Features

The behavior of the following features has been standardized by the IVI Foundation. Although these are referred to as standard driver features, only simulation must supported.

  • Simulation — Required
  • State caching — Optional
  • Range checking — Optional
  • Coercion and coercion recording — Optional
  • Interchangeability checking — Optional

This following sections describe each feature in detail. For more information on any feature, see the appropriate specification on the IVI Specifications webpage.

Simulation (required)

IVI drivers are often constructed for complex instruments that are difficult to procure or that are in high demand at the end user's facility. In these cases, users must be able to test their programs without being physically connected to an instrument. IVI driver simulation makes this possible.

When an end user enables simulation in an IVI driver, the driver performs no I/O communication whatsoever. IVI drivers are free in their choice of mechanisms for returning meaningful output values to the end user, but the intent is that the user program should be able to proceed normally when calling functions on the driver. This allows the end user to exercise their test application before the actual hardware is available.

State Caching

State caching is a way to offer better performance in customer test applications. The principle of state caching is quite simple, although in practice it can be extraordinarily difficult to implement properly.

When state caching is enabled, the driver attempts to determine if new settings passed into the driver are different from the actual instrument state. This reduces I/O traffic between the driver and the instrument because fewer I/O calls are performed.

In a simple example of how state caching improves performance, consider what happens when a user passes identical values several times in a row for the frequency property on a function generator driver:

  • Without state caching, each call into the driver results in an I/O function call to the instrument.
  • With state caching, the driver detects that subsequent calls to the frequency property are redundant, and thus does not communicate with the instrument. Correspondingly, the driver "remembers" the last known value for instrument properties, and performs queries by retrieving data from computer memory, rather than by performing instrument I/O.

As with caching schemes in other types of software systems, IVI driver state caching is fraught with potential pitfalls. The chief challenge in properly implementing state caching is maintaining cache coherency. Complex instruments often have complex couplings between instrument settings, making it difficult for the driver to track whether or not settings stored in the driver cache are valid. Consequently, many instrument vendors choose not to implement state caching in the driver (Keysight typically does not support it).

Range Checking

Range checking refers to the driver's ability to determine if input values passed to the driver are within an acceptable range. End users typically use range checking during debugging, then disable it after validation to maximize performance.

Coercion and Coercion Recording

Instruments often accept only a discrete set of values for certain properties. These discrete value sets are different for different vendor's instruments. For example, a Keysight DMM may accept 10 V, 100 V, 300 V for the voltage range setting, while a Fluke DMM accepts 20 V, 200 V, and 400 V. As a result, IVI class specifications are forced to define such real-valued properties as accepting a continuous range of values, even though the underlying instrument only accepts a discrete set of values.

Following along with the DMM voltage range example, if a user sends a value of 150 V for the voltage range, the instrument will internally coerce the value to one of the accepted values, such as 300 V. To maximize interchangeability, the IVI class specifications provide guidance on how IVI drivers should coerce user requested values for specific properties.

To aid end users in building interchangeable applications, IVI drivers may implement a feature called coercion recording. If a user enables coercion recording, the IVI driver retains a log of each user value that it coerces. Standard functions in the IVI inherent capabilities provide the user with access to the driver's coercion log.

Interchangeability Checking

A program written using an IVI driver is not interchangeable if the program relies upon default settings within the instrument. This is because different instruments have different default values for many important settings. Repeatable results can only be obtained if all of the instrument settings on which an operation depends are in a user-specified state.

IVI drivers may implement a feature called interchangeability checking to ensure that driver attributes relevant to a particular operation are in a user-specified state whenever that operation is performed. All IVI instrument class specifications include class-specific sections that specify which driver attributes must be in a known state for particular operations. The driver can then track these attributes and report a warning if one or more are not set explicitly by the client program before an important operation is performed.

Standard functions in the IVI inherent capabilities provide the user with the ability to enable interchangeability checking in the driver and to retrieve any warnings that occur. Typically, this feature is only used during development and debug of an application, then are disabled during normal operation so as not to adversely impact performance.
